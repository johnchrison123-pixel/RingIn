Executive summary

Started with one master audit (5 parallel agents → 152 findings across HomeScreen, Moments, Messages, Profile, Settings, Calls, Wallet, App.js, OTA, security). Shipped fixes in batches, then re-audited with a fresh angle each round. Every round still surfaced real user-impact bugs — the 12,000+ line codebase had genuine surface area.

Process discipline

Every batch followed the same loop:

  1. Audit agent — independent agent reads the actual code, finds bugs with file:line citations
  2. Fix agent — receives explicit instructions per item, edits surgically, NO JSX, var pattern, hook order preserved
  3. Verification agent — independent agent checks each fix for scope leaks, hook order violations, regression of prior fixes
  4. Build check — full CRA build must compile clean before ship
  5. Ship — strip-build-for-apk lean APK + OTA bundle to GitHub raw + tracker PDF update + push to main

Why fresh angles each round

After round 6 ship, the audit agent kept reporting "Clean" against the same questions. So each subsequent round was given an orthogonal angle: different file-set focus (R9), different bug class (R10–R17). Each new angle found genuine bugs the previous rounds had missed. Suggests that any "no findings" result is more a sign of stale angles than a clean codebase.

Patterns we fixed at the root

Scope leaks 7 sites caught
Variable declared in one component used from a sibling component scope. Multiple types: mutedConvos declared in ChatBox used from MessagesScreen, userPosts declared in UserProfileView used from saveEditPost, props.X used in components that destructured props. Same class of bug as the original "mutedConvos is not defined" crash. We fixed each one and audit agents specifically grep for this pattern now.
Stale closures in setInterval/setTimeout/listeners 10+ sites
hangup() in CallScreen captured by 6+ stale closures (poll, 35s timer, per-min tick), localCoins read via closure inside hangup (would have undone every per-minute deduction), restrictedSet captured at channel subscribe time, etc. Fix pattern: mirror state into ref via useEffect, read ref.current inside async callbacks.
Missing .catch on Supabase promises 15+ sites
.then(r => { if (r.error) rollback }) handles RLS denial but NOT raw network reject. Pattern shipped: every Supabase call also has .catch(e => { rollback; toastError }). Caught stuck spinners in handleAuth, processPayment, saveEditProfile, save*, toggleLike, doSearch, matchAnonymous, password reset/change, follow/unfollow, mark-read, delete, ...
Optimistic UI without rollback 8+ sites
setState(optimistic); supabase.write() with no rollback path on failure. Each one was a silent data corruption: setUnreadNotif(0) without restoring on error, deletePost without snap restore, etc. Pattern shipped: snapshot before, restore on error in both .then(r => r.error) AND .catch.
Stuck spinners on network failures 5+ sites
Loading state set true → async call → on failure, never set false → user must hard-reload. Audited every "Loading..." / "Sending..." / "Processing..." state. Fix: always reset in catch.
Hardcoded "1,240" coin balance 5 sites + CallScreen prop
Coin chip showed "1,240" everywhere across screens. Each screen had its own useState(50) or hardcoded literal. Built useCoinBalance shared hook with per-user storage + Supabase realtime + cross-screen broadcast. App.js feeds real balance to CallScreen so call deduction starts from actual wallet.
Cross-tab / cross-component event leaks ~10 listeners
Every 'ringin:*' custom event verified: producer + consumer + cleanup. Window events: ringin:back, ringin:back-call, ringin:restricted-changed, ringin:muted-convos-changed, ringin:coin-balance, ringin:open-post-detail, ringin:home-open-post, ringin:incoming-call, ringin:name-changed. All have matching addEventListener / removeEventListener.

Notes on what's NOT regression-tested

The audit + verify agents specifically check for scope leaks, hook order, missing .catch, and regression of previously-shipped fixes — but they do NOT run the actual app. Verification is static (read the code) not dynamic (tap buttons). If any fix introduced a subtle runtime bug that compiles cleanly, only your real-device testing will catch it.

If you find any regression, send me the symptom (which screen, which tap, what happened) and I'll fix it surgically in the next iteration.
